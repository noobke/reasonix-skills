# Verification Before Completion — 完成前验证

## 概述

没有验证就宣称完成不是效率，是不诚实。

**核心原则：** 证据先于声称，永远如此。

## 铁律

```
没有新鲜验证证据，就没有完成声明
```

如果你在这一轮消息中没有运行过验证命令，就不能声称它通过了。

## 门禁函数

```
在声称任何状态或表达满意之前：

1. 识别：什么命令能证明这个声明？
2. 运行：执行完整命令（新鲜、完整运行）
3. 读取：完整输出，检查退出码，数失败数
4. 验证：输出是否证实了声明？
   - 如果否：用证据说明实际状态
   - 如果是：带证据做出声明
5. 只有这时：做出声明

跳过任何一步 = 撒谎，不是验证
```

## 常见失败

| 声明 | 需要 | 不够 |
|------|------|------|
| "测试通过" | 测试命令输出：0 失败 | 上轮运行、"应该能过" |
| "Lint 干净" | Linter 输出：0 错误 | 部分检查、外推 |
| "构建成功" | 构建命令：exit 0 | Linter 通过、日志看起来没问题 |
| "Bug 修好了" | 测试原始症状：通过 | 代码改了、假设修好了 |

## 红灯 — STOP

- 使用"应该"、"可能"、"似乎"
- 在验证前就表达满意（"完美！"、"好了！"）
- 信任子代理的成功报告而不验证
- "就这一次"
- 累了想赶紧结束

**任何暗示成功但没有运行验证的措辞都不能用。**

## 关键模式

```
✅ [运行测试命令] [看到: 34/34 pass] "All tests pass"
❌ "应该能过了" / "看起来没问题"
```

**回归测试 (TDD Red-Green)：**
```
✅ 写 → 跑 (pass) → 回滚修复 → 跑 (MUST FAIL) → 恢复 → 跑 (pass)
❌ "我写了回归测试"（但没有验证 red-green）
```

## 结论

验证没有捷径。运行命令。读取输出。然后才能声明结果。

不可协商。
