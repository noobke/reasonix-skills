# Test-Driven Development (TDD)

## 概述

先写测试。看它失败。写最少代码让它通过。

**核心原则：** 如果你没看到测试失败，你根本不知道它是不是测对了东西。

**违反规则的文字就是违反规则的精神。**

## 铁律

```
没有先写失败的测试，就没有生产代码
```

在测试之前写了代码？删掉。重来。

**没有例外：**
- 不要保留当"参考"
- 不要在写测试时"调整"它
- 不要看它
- 删了就是删了

从测试开始重新实现。句号。

## RED-GREEN-REFACTOR

### RED — 写失败的测试

写一个最小的测试来展示应该发生什么。

**要求：**
- 一个行为
- 清晰的名称
- 真实代码（除非不可避免才 mock）

### 验证 RED — 看它失败

```bash
npx vitest run path/to/test.test.ts
```

确认：
- 测试失败（不是报错）
- 失败消息符合预期
- 因为功能缺失而失败（不是拼写错误）

### GREEN — 最少代码

写最简单的代码让测试通过。不要加功能、不要重构其他代码、不要"改进"超出测试范围的东西。

### 验证 GREEN — 看它通过

```bash
npx vitest run path/to/test.test.ts
```

### REFACTOR — 清理

只在绿灯后：
- 消除重复
- 改进命名
- 提取辅助函数

保持测试绿灯。不要添加行为。

### 重复

下一个失败测试对应下一个功能。

## 常见借口

| 借口 | 现实 |
|------|------|
| "太简单不需要测" | 简单代码也会挂。测试只要 30 秒。 |
| "我之后补测试" | 测试立即通过啥也证明不了。 |
| "手动测过了" | 临时 ≠ 系统化。没有记录，不能重跑。 |
| "删掉 X 小时的工作太浪费" | 沉没成本谬误。留着没验证的代码是技术债务。 |
| "TDD 太教条了" | TDD 就是务实：提交前找 bug 比事后调试快。 |

## 红灯 — 停，重来

- 测试之前写了代码
- 实现之后补测试
- 测试第一次就通过
- "这只是一次例外"
- "我已经手动测试过了"

**以上全部意味着：删代码，用 TDD 重来。**
