# Systematic Debugging — 系统化调试

## 概述

随机 fix 浪费时间并产生新 bug。快速补丁掩盖根本问题。

**核心原则：** 在尝试修复之前，始终先找到根因。治标不治本是失败。

## 铁律

```
没有根因调查就没有修复
```

如果你没完成 Phase 1，就不能提修复方案。

## 四个阶段

### Phase 1: 根因调查

**在尝试任何修复之前：**

1. **仔细读错误信息** — 错误信息通常包含精确的解决方案。完整读堆栈跟踪。
2. **稳定复现** — 能可靠触发吗？如果不能复现 → 收集更多数据，不要猜。
3. **检查最近的变更** — `git diff`、最近的提交、新依赖、配置变更。
4. **多组件系统收集证据** — 在每个组件边界增加诊断日志，找到哪一层出问题。
5. **追踪数据流** — 坏值从哪里来？从调用栈向上追踪直到找到源头。

### Phase 2: 模式分析

1. **找可工作的例子** — 同一个代码库中类似的正常代码
2. **对比差异** — 列出所有不同，不管多小
3. **理解依赖** — 这个组件依赖什么？什么配置、环境？

### Phase 3: 假设和测试

1. **形成单一假设** — "我认为 X 是根因，因为 Y"
2. **最小测试** — 做最小的变更来测试假设
3. **验证后再继续** — 有效？→ Phase 4。无效？→ 形成新假设。

### Phase 4: 实现

1. **创建失败的测试用例** — 用 `/skill tdd` 写测试
2. **实现单一修复** — 解决根因，一次一个变更
3. **验证修复** — 测试通过了吗？没有其他测试被破坏？
4. **如果修复无效** — 停下来。试了几次了？如果 ≥ 3 次，质疑架构。

## 红灯 — 停止并遵循流程

- "快速修一下，之后再调查"
- "就试一下改 X 看看行不行"
- "同时改多个地方，跑测试"
- "不用测试了，我手动验证"
- "很可能是 X，让我修一下"

**以上全部意味着：停止。回到 Phase 1。**

## 现实影响

- 系统化方法：15-30 分钟修复
- 随机 fix 方法：2-3 小时的无效尝试
- 首次修复成功率：95% vs 40%
