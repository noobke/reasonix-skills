# Writing Plans — 编写实施计划

## 概述

假设执行者对你代码库的上下文一无所知，写出全面的实施计划。记录他们需要知道的一切：每个任务涉及的文件、代码、测试、需要查阅的文档、如何验证。把计划拆成一口大小的任务。DRY、YAGNI、TDD。

**开始时宣布：** "我用 writing-plans 来创建实施计划。"

**计划保存位置：** `docs/plans/YYYY-MM-DD-<feature-name>.md`

## 文件结构

在定义任务之前，先规划哪些文件将被创建或修改，以及每个文件的职责。

- 设计边界清晰、接口明确的单元。每个文件应有单一职责。
- 更小、聚焦的文件更容易理解和可靠编辑。当文件变大时，通常是职责过多的信号。
- 一起变化的文件应该放在一起。按职责拆分，不是按技术层。

## 任务粒度

**每个步骤是一个动作（2-5 分钟）：**
- "写失败的测试" — 一步
- "运行确认它失败" — 一步
- "写最少的代码让测试通过" — 一步
- "运行测试确认通过" — 一步
- "提交" — 一步

## 计划文档头

```markdown
# [功能名] 实施计划

> **目标：** [一句话描述要构建什么]
> **架构：** [2-3 句话描述方案]
> **技术栈：** [关键技术/库]
```

## 任务结构

```markdown
### Task N: [组件名]

**文件：**
- 创建: `exact/path/to/file.ts`
- 修改: `exact/path/to/existing.ts:123-145`
- 测试: `tests/exact/path/to/test.ts`

- [ ] **步骤 1: 写失败的测试**
   (完整测试代码)

- [ ] **步骤 2: 运行验证失败**
   `npx vitest run tests/path/test.ts -t "test name"`
   预期: FAIL

- [ ] **步骤 3: 写最小实现**
   (完整实现代码)

- [ ] **步骤 4: 运行验证通过**
   预期: PASS

- [ ] **步骤 5: 提交**
   `git commit -m "feat: add specific feature"`
```

## 禁止占位符

每个步骤必须包含实际内容。以下都是**计划失败**：
- "TBD"、"TODO"、"稍后实现"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"
- "为上述写测试"（没有实际测试代码）
- 描述做什么但不展示怎么做（代码步骤必须有代码块）

## 自查清单

写完计划后，用新的眼光审视：
1. **覆盖度** — 设计的每个部分都有对应任务吗？
2. **占位符扫描** — 搜 "TBD"、"TODO"、"later"、"适当的"
3. **类型一致性** — 后面的任务引用的类型/签名和前面定义的一致吗？

## 执行交接

计划完成后，告诉用户：

> "计划已保存到 `docs/plans/<filename>.md`。下一步用 `/skill executing-plans` 来分步执行。"
